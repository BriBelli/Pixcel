# Migration scan 02 — A2UI protocol + renderer

> **Scope.** `photolif/libs/a2ui-core` (framework-agnostic A2UI protocol), `photolif/libs/a2ui-react`
> (React renderer seed), and `photolif/apps/a2ui-chat` (the mature **Lit** renderer + ~35 components +
> the chat surface). Verdict map for the React-everywhere unification (FORK-1 resolved).
> **Headline:** `a2ui-core` PORTS DIRECTLY (crown-jewel protocol, already TS, framework-agnostic). The Lit
> renderer + components are a large **RECREATE-in-React** lift (~31k LOC of Lit), partially de-risked by the
> existing React seed + the Claude Design prototypes.

---

## 1. `a2ui-core` — the protocol (PORT DIRECTLY)

Pure TypeScript, zero framework or DOM dependency. The `A2UIRegistry<R, C>` is *generic over the rendered
output type* (`R = ReactElement | TemplateResult | …`) — it was explicitly designed to be shared across
React/Lit/Flutter renderers. This is exactly the crown-jewel asset the plan calls it. Move the package as-is
into the monorepo; the only work is workspace wiring and an import-alias decision.

## 2. `a2ui-react` — the React renderer SEED (REFACTOR + EXTEND, the porting backbone)

A real React renderer already exists (~955 LOC). It is the *foundation* the FORK-1 decision leans on — but
it is a **partial prototype**, not production:

- **Covers 11 builtin atoms only:** card, button, text-field, text, image, container, select, checkbox,
  progress, divider, chip. (Lit has ~12 atomic + ~24 PXC/video.)
- **Flat-mode only.** `A2UIComponentRenderer` resolves children via `findComponent(response.root)` + string
  IDs. It does **not** handle the *nested/inline-children* mode that LLM output actually uses and that the
  Lit renderer + `schema.ts` support. **This is a correctness gap, not just missing components.**
- **No PXC/video/chart/data-table/stat/tabs/accordion/alert/grid/list** components.

| Module | Verdict | Effort | Notes |
| --- | --- | --- | --- |
| `A2UIRenderer.tsx`, `context.ts`, `hooks.ts`, `registry.ts` | **REFACTOR** | S | Good bones (context + registry + hooks). Must add **nested/inline-children traversal** to match Lit + schema. |
| `components/*.tsx` (11 atoms) | **REUSE / RESTYLE** | S–M | Reuse the React wiring; **restyle to the Claude Design tokens** (these were built to the old photolif look). |
| Missing builtins (stat, tabs, accordion, alert, grid, list, data-table, chart, separator, link) | **RECREATE** | M | Port from the Lit equivalents (logic) onto the React seed + DS tokens. |

> **Strategy note.** `a2ui-react` is the *starting point* for the renderer; the Lit `a2ui/` components are the
> *behavior reference*. The Claude Design prototypes are the *visual* reference. Three sources, one target.

## 4. Prose — recommended React-port strategy

**The split is clean and favorable.** `a2ui-core` is genuinely framework-agnostic and ports directly — the
generic `A2UIRegistry<R,C>` was built for exactly this. The cost is entirely on the *renderer side*, and even
there a React seed already exists. So the work is: **(1)** lift `a2ui-core` in as-is; **(2)** grow
`a2ui-react` from prototype to the production renderer; **(3)** RECREATE the ~35 Lit components onto it, using
the Claude Design prototypes as the visual target.

**Reusable as-is (or near-as-is):**
- All of `a2ui-core` (types, registry, schema, utils, patterns). The crown jewel ports clean.
- The `a2ui-react` *architecture* (context/registry/hooks) and its 11 atom components' *logic*.
- The Lit `a2ui-renderer.ts` `switch` as the **coverage spec** (which types map to which component; the
  stat-grouping heuristic).
- The 6 patterns as the seed of the PLATE/content-style scaffold library.
- Most `services/*.ts` (plain TS, no Lit) — rebind, don't rebuild.

**Needs full rebuild (Lit → React, no mechanical path):**
- Every Lit component's markup/styles (Shadow DOM + `html`-template + `static styles` → JSX + DS-token CSS).
- The two giants — `a2ui-app` (3,375) and `a2ui-prompt-bar` (2,840) — and `a2ui-chat-message` (1,898),
  `pxc-studio` (3,047), `pxc-scene-editor` (1,289), `a2ui-settings-panel` (1,942), `a2ui-chart` (1,057).
- The PXC studio surfaces should **defer to this repo's existing `pxs-studio`** (restyle per the integration
  guide), NOT be ported from the Lit `pxc-studio`. Porting both would duplicate the studio.

**Sequencing (matches the plan's phases):**
- *First slice:* `a2ui-core` port + `a2ui-react` renderer hardened to production (nested mode + full atom set
  + chart strategy) + the chat surface (Group C) + app shell (Group D) on the DS. This is the front door.
- *P3 (breadth):* Group B scenes/storyboard/video surfaces, alongside the Python→TS backend port.

**Top 3 risks**

1. **The React renderer only does flat-mode; LLM output is nested.** `a2ui-react/A2UIRenderer.tsx` resolves
   children via `response.root` + string-ID `findComponent`, but the Lit renderer and `schema.ts` both support
   **inline nested children** (the natural LLM format). Shipping the React renderer without nested traversal
   silently renders nothing for real agent output. **Must fix in the first slice; it's the renderer's core
   contract, not a component.**

2. **PXC types are duplicated across `a2ui-core/types.ts` and this repo's `pxs-core`.** `PXSFrame`/`PXSCell`/
   `PXSAnimation` (and quality presets) exist in both. If they drift, the A2UI `pxc-*` components and the
   studio disagree on the data model — exactly the seam the unification is supposed to fuse. Decide one source
   of truth (likely `pxs-core`) and have `a2ui-core` import/re-export it. Coordinate with the `pxs-core`
   reconciliation (P0, "reconcile 2 defaults").

3. **`a2ui-chart` is 1,057 LOC over Chart.js + 4 community plugins + topojson.** It's the single most
   entangled atom and has **no Claude Design prototype** to target. Chart.js is framework-neutral (can wrap in
   React), but the plugin/geo surface is a real port-or-replace decision. Risk of it ballooning if treated as
   "just another atom." Scope it explicitly: keep Chart.js + wrap, or trim the exotic chart types for v1.

**Open questions for the architect**

- **Import alias:** keep `@a2ui/core` / `@a2ui/react`, or rebrand to `@pxs/a2ui-*` for the one-product story?
- **PXC-type ownership:** does `a2ui-core` re-export from `pxs-core`, or do both depend on a shared types pkg?
  (Drives Risk 2.)
- **Studio overlap:** confirm the PXC studio surfaces (`pxc-studio`/`pxc-canvas`) are **restyled from this
  repo's `pxs-studio`** and the Lit versions are reference-only (TRASH-after-extract) — not a second port?
- **Chart strategy:** keep Chart.js (wrap in React) or replace? Drop the exotic chart types (sankey/funnel/
  treemap/geo) for the first slice?
- **`evaluateCondition` hardening:** the hand-rolled `when`-expression parser is fragile. Replace with a small
  safe expression lib during the port, or keep and constrain?
- **Patterns → PLATE library:** is the 6-pattern set the canonical seed for the plan's content-style/scaffold
  library, owned in `a2ui-core`? (It maps directly to the "pick-the-scaffold-then-fill-it" IP.)
